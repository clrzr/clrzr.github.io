How I think about tradeoffs

Serial vs. Parallel

Thread utilisation informs server sizing

Before choosing whether a process runs serially or in parallel, I look at what the hardware can actually support. A serial process with low throughput requirements runs fine on a single-core instance. A service with high call volume gets parallelised — and the server sized accordingly. Software and infrastructure designed together.

Async vs. Synchronous

Keep users off the critical path of system work

If validation work can happen while a user is still deciding, it should. The fraud service doesn't wait to be queried at execution time — it runs continuously in the background and emits results downstream. By execution, the answer is already cached. The user never waits for work the system could have done earlier.

Product before compliance

Good architecture anticipates regulation

The one-device policy was built because the user story demanded it. A user loses their phone, someone picks it up — automatic device invalidation on migration. The CBN formally mandated exactly this in March 2026. The system had been built that way years earlier.

Blast radius

Contain failures before they happen

PII has a single owner — one service, one path, everything routed through it. No other service queries the identity database directly. If something fails elsewhere in the system, user data is not part of the collateral. Centralising ownership is a trust decision, not a compliance checkbox.

Fintech Platform Architecture

Production GCP · AWS · Cloudflare End-to-end ownership

A data and automation company in financial services needed a complete backend platform — contactless payments, business bookkeeping, physical automation triggers, and a loan marketplace powered by real transaction data from the informal sector.

The challenge wasn't building each surface. It was making them work as a coherent system. A QR-to-door-open payment and a loan provider accessing verified transaction history — both supported by the same underlying infrastructure.

Eight distinct services, independently deployable, talking through defined contracts. PRDs, system architecture, implementation, production deployment. 99%+ uptime throughout.

The Business OS layer translates informal business activity into structured accounting data — then makes it available to loan providers as underwriting input richer than anything a credit bureau can provide.

Payments · System Design

Payment Intent Architecture

Production Idempotent by design Zero race conditions

The insight came from watching users: across every product they trusted, the destination always appeared before the amount. Not a UX pattern — a signal about how people think about money movement.

Every payment begins with a Payment Intent. The user submits their destination. The system resolves it and returns immediately with an intent ID and confirmed name. While they're reading that, validation runs in parallel.

Transaction limits, wallet balance, fraud rules — all fetched concurrently, not sequentially. By the time the user submits an amount, most of the work is already done. Near-instant execution on confirm.

One model. All payment types. Bank transfer, QR, bill payment, contactless — the frontend always deals with the same primitives. The Intent ID is the idempotency key throughout.

Payment intent flow

Key decisions

Async parallel validation during user think-time

Limits, balance, fraud rules all run while the user reads the destination name.

PIN verify + eligibility check in parallel

Two independent checks at execution. Running concurrently halves final confirmation latency.

Idempotent validation pipeline

Cached data reused. Missing data fetched. Same code path regardless of user speed.

Security · Fraud Prevention · KYC

Fraud & Identity Infrastructure

Production CBN-compliant NDPR · GDPR aligned

Most fraud systems are gates. A user tries something, the system checks a list, lets them through or blocks them. That model only knows what it's been told to look for.

This system continuously builds a picture of who the user is — their patterns, their behaviours, their normal. The question isn't just whether a transaction is suspicious. It's whether it's suspicious for this user, right now.

GPS velocity checks: location at intent creation, location at confirmation, time between them. If the distance isn't physically possible, the risk score updates immediately — catching device compromise attacks before execution.

Async KYC means provider downtime is invisible to users. Contact details are collected before verification resolves — so if something goes wrong downstream, the system can still reach them. A synchronous failure loses them silently. An async flow keeps the door open.

Key decisions

Fraud service off the critical path

Continuous background operation. Payment system honours the cached score — no override.

Single PII owner across the system

Hashed for indexing, AES-encrypted at rest. No other service queries identity data directly.
