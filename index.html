How I think about tradeoffs

Serial vs. Parallel

Thread utilisation informs server sizing

Before choosing whether a process runs serially or in parallel, I look at what the hardware can actually support. A serial process with low throughput requirements runs fine on a single-core instance. A service with high call volume and latency sensitivity gets parallelised — and the server gets sized accordingly. Software and infrastructure are designed together.

Async vs. Synchronous

Keep users off the critical path of system work

If validation work can happen while a user is still deciding, it should. The fraud service doesn't wait to be queried at execution — it runs in the background and caches results downstream. By execution time, the answer is already there. The user never waits for work the system could have done earlier.

Product before compliance

Good architecture anticipates regulation

The one-device policy wasn't built because a regulation required it. It was built because a user story demanded it. A user loses their phone. Someone picks it up. The app is still authenticated. That question had an obvious answer — and the architecture followed from it. Regulation caught up later.

Blast radius

Contain failures before they happen

PII has a single owner — one service, one access path, everything routed through it. No other service queries the identity database directly. If something fails elsewhere in the system, user data is not part of the collateral. That's an architectural decision, not a compliance checkbox.

The best architectural decisions read as product decisions in hindsight. Device safety was a product call before it was a regulation. Async KYC was a retention call before it was a technical pattern. PII isolation was a trust call before it was a compliance requirement. The ordering matters.
