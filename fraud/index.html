The broken assumption

Most fraud systems are gates. A user tries to do something, the system checks a list, and either lets them through or blocks them. That model has a fundamental weakness: it only knows what it has been explicitly told to look for. A sufficiently novel attack pattern — one that doesn't match any existing rule — passes straight through.

The second problem with the gate model is timing. By the time a user reaches a payment execution checkpoint, the system is seeing that action in isolation. It knows nothing about the login five minutes ago, the location the intent was created from, the device they're on. It makes a binary decision with almost no context.

The insight

The question a fraud system should ask isn't just is this transaction suspicious? It's is this transaction suspicious for this user, right now, given everything we know about them? That shift — from static rules to contextual understanding — changed everything about the architecture.

Instead of checking at the moment of action, the system should be continuously building a picture of who the user is. Their patterns. Their behaviours. Their normal. And every action in the system is an opportunity to update that picture — not just payment events, but logins, device changes, location shifts.

The fraud system is not a checkpoint. It is a continuously running background process that happens to inform checkpoints.

What we built

Every meaningful action in the system emits a signal to the fraud service. Not the raw data — the signal. The fact of what happened, when, and the context around it. A user logs in: the fraud service receives the login event, timestamp, device identifier, location. A user initiates a payment: the fraud service receives intent creation and location at that moment. A user confirms a transaction: location again, at confirmation.

None of this puts the fraud service in the critical path of those actions. It receives signals asynchronously, processes them, and updates its risk profile for that user in the background. As signals arrive, the profile is recalculated in real time. The service doesn't wait to be asked — it emits the updated score downstream the moment a profile changes. The payment service receives and caches it.

By the time a transaction reaches execution, the fraud service has already been watching. The risk score the payment system consults is not computed at the moment of execution. It is the result of everything that happened before.

The concrete detail — GPS velocity

When a payment intent is created, the device location is captured. When the user confirms the transaction, the location is captured again. The system evaluates the distance between those two points relative to the time elapsed between them.

If a user created a payment intent in Lagos and confirmed it from Abuja four minutes later, something is wrong. The check isn't about the distance alone — it's about whether that distance is physically possible given the time between the two events. No human moves 500km in four minutes.

This catches a specific attack pattern: device compromise where an attacker on a separate device attempts to confirm a transaction the legitimate user initiated. The legitimate user is still in Lagos. The attacker, with access to the session, is somewhere else entirely. The velocity check surfaces that discrepancy as a risk signal before execution.

The principle

The difference between a gate and this system is that a gate only knows what it was told. This system knows what it has observed. By execution time, it has seen the login. It has processed the GPS signals. It has watched the pattern of the session. The risk score isn't a spot check — it's a continuous assessment that happens to surface at the moment it's needed.

That is the difference between a fraud system that reacts and one that understands.

Key decisions

Fraud service async — never on the critical path

Synchronous fraud checks add latency to every transaction. Running continuously in the background means execution is a cache lookup, not a computation.

Cached score as primary path, live fallback if cold

Fast by default. If the cache is cold for any reason, a direct call to the fraud service is the fallback. Either way, the payment system honours the result without override.

Profile recalculated on every signal, not on request

The service doesn't wait to be polled. Every incoming signal updates the profile and emits a new score downstream. Dependent services always have the most current view.

Velocity check on location delta, not location alone

Distance without time is meaningless. The check is whether the distance between two locations is physically possible given the elapsed time — which targets compromise, not travel.
